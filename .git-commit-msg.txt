Make the ROS graph observer live and honest, not a fake pass

`run` used to write `graph_ready: pass` whenever ROBOTICS_SKIP_ROS_OBSERVER
was set, and always observed the graph only after the launched process had
already exited. Both defeated the point of the check. Now:

- graph observation happens while the process is still running, and
  `expected_ros_graph` services/actions are checked in addition to topics
- skipping the observer fails closed (`graph_ready: skip`, overall `fail`)
  instead of a silent pass; a new ROBOTICS_GRAPH_CHECK_EMBEDDED flag exists
  for launch plans (e.g. docker compose --exit-code-from) that already
  enforce readiness themselves, and is reported as derived from the process
  exit code rather than as an independent claim
- the harness's own e2e now runs a real rclpy publisher and requires a real
  publisher match (min_count: 1) instead of a min_count: 0 fixture that
  passed regardless of whether anything was actually running; CI installs
  ros-jazzy-ros-base so this is exercised for real, not skipped
- resolver.py traces list-valued overlay replacements as `list_replaced`
  distinct from scalar `replace`, with a regression test proving a
  `/clock`-dropping overlay is both traceable and fails schema validation
- CI gained a vendor schema sync gate against a pinned
  robotics-runtime-contracts ref so vendor/contracts-schemas cannot drift
  silently from the authoritative package
